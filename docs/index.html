SubDomain Hub

基于 Cloudflare Workers + D1 构建的二级域名分发系统。用户通过 GitHub 登录后提交子域名申请，经管理员审核通过后即可获得完整的 DNS 记录控制权。

快速开始 → API 文档

功能特性

🔐

GitHub OAuth 登录

仅允许 GitHub 用户申请子域名，用户身份天然可信。

✅

管理员审核制度

申请需管理员审核，支持通过/拒绝（附拒绝原因）。

📧

邮件通知

审核结果自动通过邮件通知用户，新申请通知管理员。

🌍

完整 DNS 控制

支持 A、AAAA、CNAME、MX、TXT、SRV、CAA 全类型记录。

📦

多域名支持

环境变量填写域名即可，Zone ID 自动通过 Cloudflare API 解析。

🛡️

子域名保护

内置 50+ 禁止前缀列表，防止关键子域名被滥用。

📊

配额管理

可配置每用户子域名数量和每子域名 DNS 记录数上限。

🎨

暗/亮主题

自适应主题切换，管理面板支持深色模式。

⚡

完全无服务器

运行在 Cloudflare Workers 上，零服务器成本，全球边缘部署。

前置准备

在开始之前，请确保您具备以下条件：

  • Cloudflare 账号，并且至少有一个域名托管在 Cloudflare
  • GitHub OAuth App（下一步创建）
  • Node.js 18+ 环境
  • 安装了 Wrangler CLI

安装依赖

bash# 克隆仓库
git clone <repo-url>
cd cloudflare_subdomain_provisioning

# 安装依赖
npm install

创建 GitHub OAuth App

  1. 前往 GitHub Developer Settings

    访问 github.com/settings/developers，点击 New OAuth App。

  2. 填写应用信息

    字段值
    Application nameSubDomain Hub
    Homepage URLhttps://your-worker.workers.dev
    Authorization callback URLhttps://your-worker.workers.dev/auth/github/callback

  3. 保存 Client ID 和 Client Secret

    创建完成后记下 Client ID 和 Client Secret，稍后配置需要使用。

创建 Cloudflare API Token

  1. 前往 API Tokens 页面

    访问 Cloudflare API Tokens。

  2. 创建自定义 Token

    设置以下权限：

    • Zone > DNS > Edit — 用于管理 DNS 记录
    • Zone > Zone > Read — 用于自动解析 Zone ID

    区域资源选择对应的域名（或所有域名）。

注意

必须同时包含 Zone > Zone > Read 权限，否则系统无法自动解析域名对应的 Zone ID。

创建 D1 数据库

bash# 创建数据库
npx wrangler d1 create subdomain-db

将返回的 database_id 填入 wrangler.toml 中对应位置：

toml[[d1_databases]]
binding = "DB"
database_name = "subdomain-db"
database_id = "your-database-id-here"

然后运行数据库迁移：

bash# 本地开发时
npm run db:migrate:local

# 远程（部署后）
npm run db:migrate:remote

配置环境变量

bash# 复制示例配置
cp .dev.vars.example .dev.vars

编辑 .dev.vars 文件：

env# 必需配置
GITHUB_CLIENT_ID=your_client_id
GITHUB_CLIENT_SECRET=your_client_secret
JWT_SECRET=随机生成的32位以上字符串
CF_API_TOKEN=your_cloudflare_api_token
DOMAINS=example.com,example.org
ADMIN_USERS=your_github_username
提示

DOMAINS 支持多域名配置（逗号分隔）。Zone ID 将通过 Cloudflare API 自动解析，无需手动填写。

本地开发

bashnpm run dev

访问 http://localhost:8787 即可看到应用界面。

部署到 Cloudflare

设置 Secrets

bash# 必需 Secrets
npx wrangler secret put GITHUB_CLIENT_ID
npx wrangler secret put GITHUB_CLIENT_SECRET
npx wrangler secret put JWT_SECRET
npx wrangler secret put CF_API_TOKEN
npx wrangler secret put DOMAINS
npx wrangler secret put ADMIN_USERS

# 可选: SMTP Secrets
npx wrangler secret put SMTP_HOST
npx wrangler secret put SMTP_PORT
npx wrangler secret put SMTP_USER
npx wrangler secret put SMTP_PASS
npx wrangler secret put SMTP_FROM
npx wrangler secret put SMTP_FROM_NAME

执行部署

bash# 部署 Worker
npm run deploy

# 远程数据库迁移
npm run db:migrate:remote
提示

部署完成后，Worker 将运行在 https://your-worker.workers.dev。如需自定义域名，可在 Cloudflare Dashboard 中配置 Custom Domains。

邮件通知

系统支持两种邮件发送方式，按优先级自动选择：

1. 外部 SMTP（推荐）

配置 SMTP_HOST、SMTP_USER、SMTP_PASS 后启用。由于 Workers 无法直接使用 TCP Socket，此处的 SMTP_HOST 应为 HTTP API 端点（如 Resend、Mailgun 等服务）。

envSMTP_HOST=https://api.resend.com/emails
SMTP_USER=resend
SMTP_PASS=re_xxxxxxxxxxxx
SMTP_FROM=noreply@example.com
SMTP_FROM_NAME=SubDomain Hub

2. MailChannels（默认）

不配置 SMTP 时，自动使用 MailChannels 免费 API 发信。需要在域名 DNS 中添加 SPF 记录：

dnsTXT  @  "v=spf1 a mx include:relay.mailchannels.net ~all"

API 概览

所有 API 路径以 /api 开头，需要 Cookie 认证（通过 GitHub 登录后自动获取）。返回格式均为 JSON。

认证方式

登录后 JWT 令牌自动存储在 session Cookie 中（HttpOnly + Secure + SameSite=Lax），有效期 7 天。API 请求会自动携带 Cookie，无需手动传递。

子域名 API

方法路径说明
GET/api/subdomains获取当前用户的子域名列表
POST/api/subdomains申请新子域名（进入审核队列）
DELETE/api/subdomains/:id删除子域名（含全部 DNS 记录）

子域名格式要求

  • 仅允许小写字母 a-z、数字 0-9 和连字符 -
  • 不能以连字符开头或结尾
  • 最少 2 个字符，最多 63 个字符
  • 不能使用被禁止的前缀（如 www、ns1、mail 等）

DNS 记录 API

前提条件

DNS 记录 API 仅在子域名状态为 approved（已通过审核）时可用。

方法路径说明
GET/api/subdomains/:id/records获取子域名的 DNS 记录列表
POST/api/subdomains/:id/records创建新 DNS 记录
PUT/api/subdomains/:id/records/:recordId更新 DNS 记录
DELETE/api/subdomains/:id/records/:recordId删除 DNS 记录

支持的记录类型

类型说明特殊要求
AIPv4 地址有效的 IPv4 格式
AAAAIPv6 地址有效的 IPv6 格式
CNAME别名记录—
MX邮件交换记录必须设置 priority
TXT文本记录—
SRV服务发现记录—
CAA证书授权记录—

DNS 名称映射规则

假设子域名为 myname.example.com：

name 字段实际 DNS 记录名
@ 或留空myname.example.com
wwwwww.myname.example.com
**.myname.example.com
subsub.myname.example.com

管理员 API

权限要求

以下接口需要管理员权限。管理员通过 ADMIN_USERS 环境变量配置。

方法路径说明
GET/api/admin/pending获取待审核的子域名列表
POST/api/admin/subdomains/:id/approve审核通过子域名
POST/api/admin/subdomains/:id/reject拒绝子域名（需附 reason）
GET/api/admin/subdomains获取所有子域名
GET/api/admin/users获取所有用户列表
DELETE/api/admin/subdomains/:id强制删除任意子域名

审核流程

👤 GitHub 登录 OAuth 认证
→
📝 提交申请 status: pending
→
📧 通知管理员 自动邮件
→
🔍 管理员审核 通过/拒绝
→
✉️ 通知用户 审核结果邮件
→
🌍 管理 DNS approved 后可用

详细流程

  1. 用户通过 GitHub OAuth 登录
  2. 用户提交子域名申请 → 状态变为 pending
  3. 管理员收到邮件通知（如已配置邮件）
  4. 管理员在管理面板审核：
    • ✅ 通过 → 用户收到邮件，可开始管理 DNS 记录
    • ❌ 拒绝 → 用户收到邮件（含拒绝原因）
  5. 被拒绝的用户可删除申请后重新提交
配额说明

配额计算只统计 pending 和 approved 状态的子域名。被拒绝（rejected）的不计入配额，删除后即可重新申请。

安全机制

🍪

安全 Cookie

JWT 存储在 HttpOnly + Secure + SameSite=Lax Cookie 中，前端 JS 无法读取。

🔑

OAuth State 校验

每次 OAuth 发起时生成随机 UUID 作为 state，回调时严格验证防止 CSRF。

🛡️

权限隔离

普通用户只能管理自己的子域名和 DNS 记录，管理员接口有额外中间件保护。

✅

输入校验

子域名格式正则校验、IP 地址验证、禁止前缀检查，防止恶意输入。

认证流程

text用户 → /auth/github → GitHub OAuth → /auth/github/callback → JWT签名 → Set-Cookie
                                                                              ↓
                                                          后续请求自动携带 Cookie → 验证JWT → 查询用户

中间件层级

中间件作用应用范围
authMiddleware验证 JWT，加载用户信息所有 /api/*
adminMiddleware验证管理员权限/api/admin/*
optionalAuthMiddleware可选认证（登录则加载用户）页面路由
