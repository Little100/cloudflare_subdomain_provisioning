SubDomain Hub

在 Cloudflare 边缘运行的极简二级域名分发系统。
依托 Workers 与 D1 数据库，提供免服务器部署的 GitHub OAuth 托管方案。

快速开始部署 查看 API 设置

功能特性

零服务器依赖

代码直接托管并在 Cloudflare 全球边缘节点上运行。没有中心化服务器的维护成本和风险。

GitHub 认证

舍弃了传统的注册与密码机制。用户通过 GitHub OAuth 登录，保证了身份的基础信用与可溯源度。

审批与管控

域名申请内置待审核（Pending）机制，需管理员后台手动核准。同时自带完整的保留前缀限制名单。

多类型 DNS

申请通过后，用户可自由管理 A, AAAA, CNAME, TXT, MX 等 DNS 记录，通过 Cloudflare API 实时生效。

技术栈与架构

该项目核心由轻量级组件搭建，非常适合作为开源共享基础设施运行。

  • 路由与框架：使用 Hono。得益于其轻量级特性，冷启动时间几乎为 0。
  • 持久化存储：Cloudflare D1。原生的边缘 SQLite，负责存储用户数据、域名记录与状态。
  • 认证与会话：标准的 JWT。依靠 HttpOnly 与 Secure 属性保障跨站请求伪造 (CSRF/XSS) 的安全性。
  • 交互界面：为了减少构建复杂度，后台 UI 采用原生 HTML/JS，无需 React 或 Vue 执行复杂渲染。

快速部署

推荐采用「从 GitHub 连接」的方式部署本应用，这意味你无需在本地安装 Node.js 或者设定复杂的开发环境，一切都在云端进行。

1. Fork 仓库

前往 本项目的开源仓库页，将仓库 Fork 至你的 Github 个人账号下。

如需变更内置规则（如限制单人注册量、保留前缀黑名单等），可直接在仓库页直接修改 wrangler.toml 的 [vars] 部分。

2. 创建 OAuth 应用

登录 Github，前往 Settings > Developer Settings > OAuth Apps 新建应用。

  • Homepage URL: https://your-worker-name.your-account.workers.dev
  • Authorization callback URL: https://your-worker-name.your-account.workers.dev/auth/github/callback

获取 Client ID 及 Client Secret，妥善保存备用。

3. 获取 Cloudflare Token

在 Cloudflare 控制台 (API Tokens) 为其准备特定权限的 Token 用以管理 DNS 区域：

  • 需要 Zone.DNS.Edit (创建与删除 DNS 记录)
  • 需要 Zone.Zone.Read (必须 - 用于寻址自动匹配对应的 Zone ID)
4. 连接与绑定 D1 数据库

在 Cloudflare 控制台 新建 D1 数据库并命名为 subdomain-db。记录下对应的 Database ID。

此时进入 Workers & Pages 新建一个项目，选择 Connect to Git 并选择刚 Fork 的仓库，在此项目的配置界面找到 Bindings，将其与刚创建的 D1 数据库绑定，变量名为默认的 DB。

公开环境变量

此类变量不涉及直接安全信息，统一位于 wrangler.toml 文件的 [vars] 块中定义，如若修改重新 Push 即可：

变量名 默认值 功能描述
BANNED_PREFIXES www,ns1,mail... 逗号分隔的系统保留前缀
MAX_SUBDOMAINS_PER_USER 1 单用户申请额度（防滥用控制）
MAX_RECORDS_PER_SUBDOMAIN 20 单一子域名允许设置的最大解析条数

敏感凭据 (Secrets)

涉密信息必须在 Cloudflare 的 Settings > Variables and Secrets 页面创建为类型为 Secret 的环境变量（以确保加密安全）：

Secret 键名 填入信息 必要性
GITHUB_CLIENT_IDGitHub OAuth ID核心组件
GITHUB_CLIENT_SECRETGitHub OAuth 密匙核心组件
JWT_SECRET自行生成的加解密哈希长度段核心组件
CF_API_TOKEN预先签发好的 Cloudflare API 令牌核心组件
DOMAINS可用域名池。例如 example.com,io.net核心组件
ADMIN_USERS首个管理员的 GitHub 登录用户名系统管理必须
关于 JWT_SECRET

它是用户会话凭证签名的依据。推荐在终端环境下使用 openssl rand -hex 32 随即生成足够复杂的密钥段并填入。

数据库初始化（数据迁移）

部署完成后项目已在线，核心步骤是完成建表结构的初始化：

进入控制台执行 SQL

回到 Cloudflare 控制台的 D1 面板中进入 subdomain-db 的 Console。

将项目中 migrations/0001_init.sql 的建表语句全量拷贝进去并执行（这包含了 users、subdomains 和 dns_records 以及对应的索引）。

至此，整个服务端初始化工作结束，均可以在前台访问了。

API 接口与认证

该服务支持前后端分离或自定义控制面板对接。全局接口端点设于 /api/*。所有的业务鉴权依赖于请求中携带的 Cookie。

基于 Hono 编写了统一前置中间件拦截请求，当检测有效 Cookie 且在有效期内才会放行放权给业务层代码。

核心节点（用户端）

GET /api/me 获取自身的 GitHub 用户资料、注册时间与 Admin Flag
GET /api/domains 列出现供申请的核心域名集与限制规范（如限制前缀列表）
POST /api/subdomains 提交全新的子域名申请（参数: subdomain / domain）
POST /api/subdomains/:id/records 对于已获批的二级域名，提交新的 CNAME 或 A 记录

核心节点（管理端）

包含 /api/admin/* 的所有端点，中间件将查验用户 is_admin 标识。如果为 0 或未识别均将返回无权限 HTTP 403 错误。

GET /api/admin/pending 提取当前待处理排队的所有二级域申请
POST /api/admin/subdomains/:id/approve 批准对应 ID 的申请通过
POST /api/admin/subdomains/:id/reject 驳回申请并附加文字型拒绝理由（需载入 reason 参数）
